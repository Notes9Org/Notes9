# Notes9 Quickstart Guide

## Purpose

Use this guide to bring a team into Notes9 without trying to model every workflow on day one. The goal is to set up one real, active workflow so people can see how projects, experiments, notes, and literature connect in practice.

## Who This Is For

- Lab managers coordinating structure and rollout
- Researchers who need a first working flow
- Pilot teams evaluating Notes9 against current lab operations

## Recommended First Workflow

Start with a workflow that already creates friction today. Good examples:

- A project with multiple experiments and repeated handoffs
- A literature-heavy effort where evidence gets lost between papers and notes
- A reporting workflow that requires rebuilding context every week

Avoid broad migration at the start. One live workflow is enough to evaluate whether the model is useful.

## Setup Sequence

### 1. Create the Project Layer

Create one project and fill in:

- project objective
- owner
- timeline or current phase
- short working description

Use the project as the top-level container for all related experiments, notes, and literature.

### 2. Add the Core Experiment Record

For the first experiment:

- create the experiment inside the correct project
- add its purpose and expected outcome
- link the protocol if one already exists
- capture relevant materials or files at the experiment level

The experiment should answer one question clearly: what work is being done and why?

### 3. Capture a Real Lab Note

Open an existing experiment and document real work in the note editor. Capture:

- what happened
- observations
- deviations from protocol
- decisions made during execution
- references or papers that informed the work

Do not wait for perfect formatting. The first goal is a complete, connected record.

### 4. Add Supporting Literature

Use literature search to find the papers that matter to the active workflow. Save only the papers that influence:

- experiment design
- interpretation
- reporting
- next-step planning

Add relevance notes so future readers know why the paper was kept.

### 5. Test Retrieval

Before expanding usage, test whether the setup helps the team answer:

- Why was this experiment run?
- What evidence supported the design?
- What happened during execution?
- What should a different team member read first?

If those answers are easier to recover than before, the workflow is working.

## Roles and Responsibilities

### Lab Lead or PI

- approves the first workflow to model
- confirms naming and documentation expectations
- reviews whether the records are decision-ready

### Team Admin or Operations Lead

- creates the initial structure
- ensures workspace conventions are followed
- verifies access and team onboarding

### Researcher or Scientist

- records the work as it happens
- attaches evidence to the relevant workflow object
- flags ambiguity instead of leaving silent gaps

## Minimum Standards

For a pilot to stay useful, require the following:

- projects have a clear objective
- experiments have a stated purpose
- notes capture what actually happened
- saved literature includes relevance context
- handoffs refer back to the Notes9 record instead of separate memory

## What Success Looks Like

After one to two weeks, the team should be able to:

- recover prior context without asking the original author
- link decisions back to evidence
- produce a cleaner status update from the stored record
- identify missing structure before it becomes a reporting problem

## Common Mistakes

- creating too many projects before teams adopt one
- documenting only final results and not reasoning
- storing files without context
- using AI outputs without reviewer judgment
- treating Notes9 as a passive archive instead of an active workflow tool
