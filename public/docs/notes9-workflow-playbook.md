# Notes9 Workflow Playbook

## Purpose

This playbook explains how to use Notes9 as an operating layer across the research cycle. It is intended for teams that want consistent usage patterns rather than ad hoc note capture.

## Core Principle

Every important piece of work should remain attached to the workflow that produced it.

That means:

- papers stay linked to the project or experiment they inform
- notes stay linked to the work being executed
- outputs stay linked to the evidence and decisions behind them

## Workflow Model

### Projects

Projects are the top-level operating container. Use them for:

- research goals
- milestone tracking
- experiment grouping
- ownership and accountability

Create a new project only when the objective, team, or reporting boundary is meaningfully different.

### Experiments

Experiments hold execution context. Use them for:

- purpose
- protocol linkage
- operational records
- associated files or materials
- experiment-specific notes

An experiment should be understandable without reading separate email or chat threads.

### Lab Notes

Lab notes are the narrative record of work. Good notes contain:

- objective for the session
- what was done
- observations
- deviations or unexpected events
- next actions

Use one note to reflect one coherent unit of work. Do not mix unrelated sessions into the same note.

### Literature Reviews

Literature review is for evidence staging and retrieval. Use it to:

- search relevant papers
- save high-value sources
- record why each source matters
- connect evidence to active decisions

Do not save papers without context. A citation with no reason for inclusion becomes dead inventory.

## Recommended Working Pattern

### Before Work Starts

1. Confirm the project is the correct container.
2. Open or create the experiment that matches the planned work.
3. Review the literature or prior notes that inform the session.
4. Make the expected outcome explicit.

### During Execution

1. Record observations in the note editor while work is fresh.
2. Document deviations at the time they occur.
3. Attach supporting files to the experiment, not to personal storage only.
4. Use citations when specific claims or rationale need evidence.

### After Execution

1. Write a short interpretation of the result.
2. Capture open questions and next actions.
3. Link the outcome back to the project context.
4. Prepare reporting from the stored record rather than memory.

## Naming Guidance

Use stable, predictable naming conventions.

### Projects

Format:

`Program or objective - workstream - time period`

Example:

`Apoptosis signalling - BH3 profiling - Q1 2026`

### Experiments

Format:

`Question or assay - condition or cohort - date`

Example:

`Caspase activation assay - inhibitor panel - 2026-01-14`

### Notes

Format:

`Experiment name - session summary - date`

Example:

`Caspase activation assay - replicate readout review - 2026-01-14`

## Reporting Pattern

When preparing updates, work from the record in this order:

1. experiment purpose
2. note summary
3. linked evidence
4. files or outputs
5. next action

This reduces the risk of reporting conclusions without the supporting trail.

## AI-Supported Steps

AI can accelerate:

- summarization
- citation drafting
- literature triage
- formatting and rewriting

AI should not replace:

- scientific judgment
- approval of conclusions
- interpretation of ambiguous evidence
- governance decisions

## Operational Checklist

- each active project has a clear owner
- each experiment states why it exists
- each important session has a note
- key papers include relevance comments
- reporting can be reconstructed from Notes9 alone

## Escalation Triggers

Review the workflow design if any of these appear repeatedly:

- duplicate projects for the same objective
- notes with no experiment linkage
- saved papers with no explanation
- AI text copied into records without review
- teams still relying on side documents for basic context
